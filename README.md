# CS 255 Portfolio Reflection

## DriverPass Project Summary

The DriverPass project focused on designing a system for a client named DriverPass. DriverPass wanted a web based system that would help customers prepare for driving tests by allowing them to register for accounts, purchase training packages, schedule driving lessons, take online practice tests, and track their progress. The system also needed to support employees such as the secretary, trainers, owner, and IT officer with scheduling, reporting, role based access, account management, and DMV content updates. My business requirements document focused on identifying what the client needed from the system, while my system design document translated those needs into UML diagrams, technical requirements, and design decisions. :contentReference[oaicite:0]{index=0} :contentReference[oaicite:1]{index=1}

## What I Did Particularly Well

I think I did particularly well at connecting the client’s business needs to the actual system design. I made sure the system included the major functions DriverPass requested, such as online registration, package management, lesson scheduling, practice testing, progress tracking, and reporting. I also focused on security and access control because the system would handle customer information, payments, schedules, and employee permissions. In the system design document, I used multiple UML diagrams to show the system from different viewpoints, including use case, activity, sequence, and class diagrams. This helped make the design clearer for both technical and nontechnical stakeholders.

## What I Would Revise

If I could revise one part of my work, I would improve the diagrams even more by making them more detailed and easier to read. For example, I would make sure every workflow clearly shows alternate paths, failed actions, and system responses. In the scheduling process, I would continue improving how the system handles unavailable time slots, package hour limits, cancellations, and audit logging. Making these details clearer would help developers understand exactly how the system should behave before implementation begins.

## Interpreting User Needs

I interpreted the user’s needs by reviewing what DriverPass wanted the system to accomplish and turning those needs into functional and nonfunctional requirements. For example, customers needed a simple way to schedule lessons and take practice tests, while employees needed tools for managing schedules, accounts, reports, and training progress. I implemented these needs into the design by including role based permissions, a responsive web interface, secure login, DMV content updates, scheduling services, reporting tools, and database support. Considering the user’s needs is important because software should solve the actual problem the client and users are facing. If the design ignores the users, the final system may be technically functional but still frustrating, incomplete, or ineffective.

## My Approach to Software Design

My approach to designing software is to first understand the client’s goals, users, and problems before thinking about the technical solution. I like to break the system into requirements, user roles, workflows, and data objects. From there, I can use diagrams such as use case diagrams, activity diagrams, sequence diagrams, and class diagrams to organize the design. In the future, I would continue using interviews, requirement lists, UML diagrams, user stories, and iterative feedback to analyze and design systems. I would also focus more on testing the design against real user scenarios so that the final system is practical, secure, and easy to use.
